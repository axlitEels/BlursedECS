## Неизбежные Вопросы

### Указатели? Где? Зачем?

Указатели использовал исключительно для хранения полиморфных классов, а именно у меня заведены вектора уникальных указателей на IComponentStorage и System. Первый нужен, чтобы хранить все компоненты в одной кучке. Второй же нужен, чтобы хранить функторы с какими-либо полями данных. Примеры, вроде, наглядные получились.

Компоненты я храню не по указателю, про что расскажу ниже.

### Почему я храню компоненты в `deque<optional<Component>>`?

Существуют различные библиотеки, которые предоставляют замену для `std::optional` с нулевой стоимостью по памяти, где для отслеживания пустого состояния используется нелегальное состояние содержимого. Автор библиотеки [tiny-optional](https://github.com/Sedeniono/tiny-optional/) приводит список похожих библиотек, из которых можно было бы выбрать более подходящую под данный случай. Зная о возможности оптимизировать код в будущем, я реализовал свою библиотеку на `std::optional`, чтобы не перегружать код передачей информации о sentinel value.

Итак, у нас есть "бесплатные" опциональные значения, и мы о них помним. Тестовое задание намекает на хранение компонентов в векторе указателей. Разыменование вектора и разыменование умного указателя - это две операции разыменования - цена доступа к компоненту при использовании такого решения. Согласно документации, доступ к содержимому deque тоже осуществляемся за два разыменования, так как вместо очевидной реализации через связный список в STL для хранения объектов двухсторонней очереди используется последовательность массивов фиксированной длины. По доступу к индексу, эти реализации примерно равны.

Почему я предпочел deque? Если не исключать элементы из deque, это крайне практичный контейнер. В то время, как вектор при переполнении переносит всё содержимое в новый участок памяти, двухсторонняя очередь просто аллоцирует новый отрезок за константное время. Вдобавок, двухсторонняя очередь опциональных значений решает другую важную проблему: когда мы удаляем компонент, хранящийся в умном указателе, мы деаллоцируем память, а когда добавляем новый компонент - аллоцируем её вновь. В двухсторонней очереди опциональных типов мы можем инвалидировать значение компонента. а потом переиспользовать этот участок памяти при появлении нового компонента этого же типа. Для чего я использую очередь вакантных позиций.

Использовать этот подход с вектором (чтобы было `vector<optional<T>>`) так же мешало то, что когда вектор будет расширяться, он будет перемещать все компоненты. Во-первых это ломает ссылки, а во-вторых стоимость этого переноса зависит от размера компонента, который разработчик библиотеки ограничивать не должен.

### Почему реализации шаблонов в .hpp, а не выведены в отдельный файл, например .tpp?

У меня не получилось найти решения, при котором такой подход к организации кода совместим с clangd. Если по совету в интернете подключать заголовки в файлах реализации шаблонов, то clangd жалуется на невозможность сгенерировать преамбулу из-за рекурсивных включений, а если не подключать - то он не знает о символах, определяемых в заголовках. [Однако](https://github.com/clangd/clangd/issues/811).
